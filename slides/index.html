J1.03
Le problème du Big Data

Quand une seule machine ne suffit plus, il faut changer de stratégie.

Les 3 V

Volume

Des téraoctets à des pétaoctets. Un mois de taxi NYC : 3 millions de lignes. Une année : 12 fichiers.

Vélocité

Des données qui arrivent en continu : flux, capteurs, logs, événements temps réel.

Variété

CSV, JSON, Parquet, schémas imbriqués, données semi-structurées à unifier.

Scale-up contre scale-out

Scale-upScale-out
Une machine plus grossePlus de machines
Plus de RAM, plus de CPUTravail réparti sur le cluster
Limite physique et coûtExtensible horizontalement
Un point de panneTolérant aux pannes
Le mur : passé une certaine taille, on ne peut plus acheter de machine assez grosse. Spark choisit le scale-out : répartir le calcul sur plusieurs machines.

J1.05
Une brève histoire de Spark

De MapReduce au standard de l'industrie en moins de vingt ans.

2004 - MapReduceGoogle publie le modèle map puis reduce sur disque
2006 - HadoopImplémentation open source du modèle, HDFS + MapReduce
2009 - SparkAMPLab de Berkeley, Matei Zaharia : calcul en mémoire
2013 - ApacheDon à la fondation Apache, communauté mondiale
Aujourd'huiStandard de fait : Databricks, cloud, Delta Lake
Le déclic : MapReduce écrit sur disque entre chaque étape. Spark garde les données en mémoire entre les opérations, ce qui change tout sur les traitements itératifs et interactifs.
J1.06
Pourquoi Spark a gagné

En mémoire et orchestré par un graphe d'exécution, pas un aller-retour disque permanent.

Hadoop MapReduce

Lire (disque)
->
Map, écrire (disque)
->
Reduce, écrire (disque)

Le disque est sollicité entre chaque étape. Sur un algorithme itératif, c'est le goulot d'étranglement.

Spark

Lire (disque)
->
Transformations en mémoire (DAG)
->
Écrire seulement au besoin

Spark construit un graphe d'exécution (DAG) et garde les données intermédiaires en RAM.

Résultat : jusqu'à 10 à 100 fois plus rapide que MapReduce sur les traitements itératifs (machine learning, requêtes répétées). Et une API bien plus agréable que d'écrire des map et reduce à la main.

J1.09
Partitions et parallélisme

Une partition, une unité de travail, une task. C'est l'unité de base du parallélisme.

PartitionUn bloc de lignes du jeu de données, traité indépendamment.
TaskLe calcul appliqué à une partition, sur un cœur.
ParallélismeAutant de tasks simultanées que de cœurs disponibles.
Trop peuDes cœurs inactifs, on n'exploite pas la machine.
TropDes micro-tâches : le pilotage coûte plus que le calcul.
Bon équilibreQuelques partitions par cœur, charge répartie uniformément.
Observer le nombre de partitions : df.rdd.getNumPartitions(). On réglera ce nombre finement au jour 3 (repartition contre coalesce).
J1.10
Le RDD, la brique de base

Resilient Distributed Dataset : la collection distribuée historique de Spark.

Immutable

On ne modifie pas un RDD : chaque transformation en crée un nouveau. Plus simple à raisonner et à paralléliser.

Distribué

Les données sont réparties en partitions sur les executors, jamais centralisées sur une seule machine.

Évaluation paresseuse

Les transformations ne s'exécutent pas tout de suite. Spark attend une action pour lancer le calcul.

Tolérant aux pannes

Grâce au lineage, une partition perdue est recalculée, sans réplication coûteuse des données.

En pratique : on préfère le DataFrame, plus haut niveau et optimisé (vu en fin de journée). Mais comprendre le RDD, c'est comprendre comment Spark fonctionne réellement en dessous.
J1.11
Transformations contre actions

Les transformations décrivent le calcul, les actions le déclenchent.

Transformations (lazy)Actions (déclenchent)
mapcollect
filtercount
flatMaptake(n)
reduceByKeyfirst
distinctsaveAsTextFile
Règle d'or : rien ne s'exécute avant une action. On peut empiler dix transformations sans calcul, puis tout se déclenche au premier count().

Word count sur du texte

sc = spark.sparkContext
lignes = sc.textFile("data/datasets/notes.txt")

# Transformations : aucun calcul ici
mots = (lignes
        .flatMap(lambda l: l.split())
        .map(lambda m: (m.lower(), 1))
        .reduceByKey(lambda a, b: a + b))

# Action : tout se déclenche maintenant
top = mots.takeOrdered(5, key=lambda kv: -kv[1])
print(top)

Le DAG complet n'est construit et exécuté qu'au moment du takeOrdered.

J1.12
Lineage et tolérance aux pannes

Spark ne réplique pas les données : il garde la recette pour les recalculer.

1. LineageSpark mémorise la suite de transformations
2. PanneUn executor tombe, une partition est perdue
3. DétectionLe Driver constate la partition manquante
4. RecalculSeule cette partition est rejouée depuis le lineage
5. RepriseLe job continue, sans tout recommencer
L'avantage : pas besoin de répliquer chaque donnée à plusieurs endroits. On garde la recette (le lineage), bien moins coûteuse que les données elles-mêmes. C'est le R de RDD : Resilient.
J1.13
Du RDD au DataFrame

Ajouter un schéma débloque l'optimisation automatique. C'est ce qu'on utilise tous les jours.

RDD - bas niveau

  • Une collection d'objets Python opaques pour Spark
  • Aucune optimisation : Spark exécute vos lambdas telles quelles
  • Pas de schéma, pas de types connus à l'avance
  • Code verbeux, sujet aux erreurs
Réservé aux cas particuliers : traitement très bas niveau ou structures non tabulaires.

DataFrame - haut niveau

  • Une table distribuée avec un schéma (colonnes typées)
  • Catalyst : optimise le plan logique puis physique
  • Tungsten : gestion mémoire et exécution efficaces
  • API expressive, proche de SQL
df = spark.read.parquet(
    "data/datasets/yellow_tripdata_2024-01.parquet")

(df.select("PULocationID", "fare_amount")
   .filter(df.fare_amount > 0)
   .groupBy("PULocationID")
   .avg("fare_amount")
   .show(5))

J2.02
Sélectionner et filtrer

select, filter / where, col, expr, alias : les transformations de base

Code PySpark

from pyspark.sql import functions as F

courses = (
    df.select(
        F.col("PULocationID").alias("zone_depart"),
        "trip_distance",
        F.col("total_amount"),
        F.expr("tip_amount / total_amount * 100")
            .alias("pct_pourboire"),
    )
    .filter(F.col("trip_distance") > 0)
    .where("total_amount > 0")
)

courses.show(5, truncate=False)

À retenir

  • select : projette les colonnes voulues (et en dérive de nouvelles).
  • filter et where sont strictement équivalents.
  • F.col("x"), df.x ou la chaîne "x" : trois façons de désigner une colonne.
  • alias renomme une colonne dérivée.
  • expr permet d'écrire une expression SQL en ligne.
Lazy : tant qu'on enchaîne des transformations, rien ne tourne. Seul show déclenche le calcul.
J2.03
Créer et transformer des colonnes

withColumn, when / otherwise, cast : enrichir les courses taxi

Durée, prix au km, catégorie

enrichi = (
    df
    .withColumn("duree_min",
        (F.col("tpep_dropoff_datetime").cast("long")
         - F.col("tpep_pickup_datetime").cast("long")) / 60)
    .withColumn("prix_par_km",
        F.round(F.col("fare_amount")
                / (F.col("trip_distance") * 1.60934), 2))
    .withColumn("type_course",
        F.when(F.col("trip_distance") < 2, "courte")
         .when(F.col("trip_distance") < 8, "moyenne")
         .otherwise("longue"))
)

Points clés

withColumn

Ajoute une colonne, ou la remplace si le nom existe déjà. Chaîne plusieurs appels.

when / otherwise

Le CASE WHEN de Spark : catégoriser sans UDF.

cast

Convertir un type : un timestamp en long (secondes) pour calculer une durée.

Piège : diviser par trip_distance nul donne null ou l'infini. Filtrer les distances nulles avant (voir J2.14).

J2.09
Pourquoi Parquet change tout

Colonnaire, compressé, auto-décrit et filtrable à la source

Projection de colonnes

Sélectionner 3 colonnes sur 20 ne lit que ces 3 colonnes sur le disque.

Compression colonnaire

Des valeurs du même type côte à côte se compressent très bien (souvent x5 à x10 contre du CSV).

Schéma embarqué

Les types sont dans le fichier : aucune inférence, lecture immédiate et fiable.

Predicate pushdown

Un filtre trip_distance > 10 est appliqué à la lecture : Spark saute les blocs hors plage.

En pratique : sur le fichier taxi, un select de 3 colonnes avec un filtre peut lire 10 fois moins de données qu'un CSV équivalent. Moins d'entrées / sorties, moins de réseau, plus vite.

Sur du CSV, définir le schéma soi-même est plus sûr et plus rapide

Inférence (pratique, fragile)

Nettoyer les courses taxi

propre = (
    df
    # supprimer les lignes sans zone ni montant
    .na.drop(subset=["PULocationID", "total_amount"])

Comment Spark exécute un job

Du code à l'exécution : la hiérarchie Job, Stage, Task

Action

->
Job
->
Stages
frontières de shuffle
->

Job

Déclenché par une action (count, collect, write). Aucune action, aucun job.

Stage

Un ensemble de transformations sans shuffle. Un nouveau stage commence à chaque shuffle.

Task

L'unité de travail : une task traite une partition, sur un coeur d'executor.

À retenir : nombre de tasks d'un stage = nombre de partitions en entrée. C'est le parallélisme réel.

  • select, filter, withColumn
  • map, flatMap, union

Chaque partition de sortie dépend d'une seule partition d'entrée. Aucun déplacement de données sur le réseau.

Rapide : Spark enchaîne ces opérations dans un même stage (pipelining).

Larges (wide)

  • groupBy, join, distinct
  • orderBy, repartition

Une partition de sortie dépend de plusieurs partitions d'entrée. Spark doit redistribuer : c'est un shuffle.

Coûteux : nouveau stage, écriture disque et trafic réseau entre executors.

J3.03
Le shuffle, ce qui coûte cher

Redistribuer les données par clé : disque plus réseau plus sérialisation

Mapcalcul local
Shuffle writeécriture disque
Transferttrafic réseau
Shuffle readpar clé
Reduceagrégation

Pourquoi c'est lent

  • Sérialisation des données
  • Écriture sur disque local (spill)
  • Transfert réseau entre executors
  • Tri et regroupement par clé

Comment le réduire

  • Filtrer avant de joindre ou agréger
  • Privilégier le broadcast join (J3.09)
  • Éviter distinct et orderBy inutiles
  • Bien dimensionner les partitions
Règle d'or de la perf Spark : un job lent, c'est presque toujours un shuffle de trop ou un shuffle mal réparti.

Critèrerepartition(n)coalesce(n)

J3.07
Données mal réparties (skew)

Une partition énorme ralentit tout le stage

Le problème

Une clé ultra-fréquente (ex : la zone JFK, ou un NULL) concentre les lignes. Une seule task fait tout le travail.

Task 13 s
Task 23 s
Task 33 s
Task 490 s
Symptôme Spark UI : durée max d'une task très au-dessus de la médiane (Stages, colonne Duration).

Les pistes

  • AQE skew join : Spark 3+ découpe automatiquement les partitions trop grosses
  • Salting : ajouter un suffixe aléatoire à la clé pour éclater le groupe
  • Filtrer les valeurs aberrantes ou les NULL en amont
  • Broadcast la petite table si le skew vient d'un join
# Salting : eclater une cle trop frequente
df = df.withColumn(
    "cle_sel",
    F.concat_ws("_", "PULocationID",
        (F.rand() * 8).cast("int"))
)

Broadcast join

Diffuser la petite table pour éviter le shuffle de la grande

from pyspark.sql import functions as F

courses = spark.read.parquet(
    "data/datasets/yellow_tripdata_2024-01.parquet")
zones = spark.read.option("header", True).csv(
    "data/datasets/taxi_zone_lookup.csv")

# 3 millions de courses x 265 zones
enrichi = courses.join(
    F.broadcast(zones),
    courses.PULocationID == zones.LocationID,
    "left"
)

Quand l'utiliser

Une des deux tables est petite (zones, référentiels, dimensions) et tient en mémoire d'un executor.

Gain : la grande table n'est plus redistribuée. On supprime un Exchange complet.
Auto sous autoBroadcastJoinThreshold (10 Mo). F.broadcast() force le choix.

J3.11
Predicate pushdown et partition pruning

Ne lire que ce dont on a besoin, dès la source

Predicate pushdown

Le filtre est poussé dans le lecteur Parquet. Spark saute les blocs (row groups) hors plage grâce aux statistiques min/max.

df = spark.read.parquet(
    "data/datasets/yellow_tripdata_2024-01.parquet")

# Le filtre descend jusqu'au scan
df.filter(F.col("trip_distance") > 10).count()
# Plan : PushedFilters: [GreaterThan(trip_distance,10)]

Partition pruning

Sur des données partitionnées par dossier, Spark ne lit que les partitions concernées par le filtre.

# Donnees ecrites avec partitionBy("year","month")
df = spark.read.parquet("data/out/courses_partitionnees")

# Ne lit QUE le dossier year=2024/month=1
df.filter((F.col("year") == 2024) &
          (F.col("month") == 1)).count()
Effet combiné : moins d'IO disque, moins de données à traiter, donc moins de shuffle ensuite. Tout commence par une bonne lecture.

from pyspark.ml.feature import VectorAssembler
from pyspark.ml.regression import LinearRegression

data = (df.select("trip_distance",
                  "passenger_count", "tip_amount")
          .na.drop())

assembler = VectorAssembler(
    inputCols=["trip_distance", "passenger_count"],
    outputCol="features")
prepare = assembler.transform(data)

modele = lr.fit(prepare)
modele.transform(prepare).select(
    "tip_amount", "prediction").show(5)
